Designing with AI

Design Process

I can take a digital product from an initial sketch to a working, documented prototype while maintaining ownership of the design decisions.

Overview

I use AI to extend the range of product work I can execute without replacing design judgment. It allows me to move from an ambiguous idea to a defined product direction, refined interface, working prototype, documented system, and developer-ready specification. Throughout the process, I remain responsible for establishing the direction, defining constraints, evaluating recommendations, reviewing the implementation, and deciding what belongs in the final product.

This workflow allows me to contribute across the product design lifecycle while bringing specialist depth in visual design and design systems. In a lean product team, I can develop an idea far enough for stakeholders to evaluate it and for engineers to understand what should be built, how it should behave, and why key decisions were made.

From sketch to product direction

To gain clarity, I'll sketch to work out interactions between iterative design passes

My process often begins away from the screen, with a sketch that captures an interaction or unresolved product question. I use AI to examine the concept, identify missing states, challenge assumptions, and explore how the interface could behave. These conversations help transform an incomplete idea into a clearer product direction before I invest time in high-fidelity design or code.

I may ask AI to translate the sketch into an initial wireframe, but I do not treat the first output as the solution. I compare it with the original intent, remove unnecessary complexity, refine the hierarchy, and redirect the work when the interface begins solving the wrong problem.

Capabilities:

  • Sketch interpretation

Designing through conversation

Working effectively with an AI agent requires more than writing a single prompt. I establish the product context, document what has already been decided, define constraints, and provide enough information for the agent to reason within the project. I then steer the conversation through targeted questions, corrections, comparisons, and increasingly specific decisions.

I use these conversations for discovery, requirements definition, design critique, state planning, accessibility, technical feasibility, and architectural reasoning. I can ask the agent to defend a recommendation, expose tradeoffs, or reconsider an approach that introduces unnecessary complexity. Intentional conversation becomes part of the design process—not a shortcut around it.

How I steer the work

  • Establish product context
  • Define clear constraints
  • Question recommendations
  • Compare alternatives
  • Protect design intent
  • Reduce unnecessary complexity

Prototyping in code

I use VS Code to edit prototypes and perform code reviews.

I use a coding agent to translate approved design direction into working HTML, CSS, and JavaScript prototypes developed in Github branches. The code allows me to evaluate behavior that static screens cannot fully communicate, including responsive layouts, interactive states, panel transitions, tooltips, animations, and data visualizations. I monitor while the agent builds, steer if needed, review in the browser, test interactions, and refine the experience and code through iterative passes.

I define the intended behavior, preserve the relevant architecture, monitor the implementation, and test each iteration in the browser. My front-end background allows me to review the structure, identify unnecessary complexity, and make targeted code edits when that is the clearest solution.

What I prototype

  • Responsive interfaces
  • Interactive behaviors
  • Component states
  • Panel transitions
  • Motion studies
  • Data visualizations
  • Internal design tools

Specifications and developer handoff

Pushing bug fixes in Github

Once an experience has been evaluated, I use AI to help convert the approved decisions into structured documentation. Depending on the product, this may include a product brief, feature specification, behavior specification, functional requirements, interaction notes, accessibility guidance, code examples, and implementation constraints.

I also use this workflow to organize Markdown documentation, prepare GitHub branches, review changes, document bug fixes, and preserve the reasoning behind important decisions. A product team receives more than a collection of screens: it receives a working model of the experience and a clear record of how the product is intended to behave.

Handoff deliverables

  • Product briefs
  • Feature specifications
  • Behavior specifications
  • Functional requirements
  • Accessibility notes
  • Code examples
  • Implementation guidance

A human-directed AI workflow

AI increases the range and speed of work I can execute, but it does not replace judgment or accountability. I remain responsible for determining which problem should be solved, whether a recommendation is appropriate, how the experience should feel, and when the work is ready to move forward.

I validate output through visual review, interaction testing, code inspection, and comparison with the established requirements. When an agent recommendation introduces more complexity than the problem requires, I reject it, redirect the implementation, or make the focused code edit myself.

Result

AI does not make the product decisions for me. It gives me a multidisciplinary partner I can direct, question, and collaborate with as I move a product from concept to execution.

Reflection

This approach allows me to operate across product definition, visual design, interaction design, coded prototyping, documentation, and early implementation. It is especially valuable for zero-to-one work and lean teams that need someone who can bring structure to an unclear idea and make the experience tangible.

“Working with AI has made my design process more technical, iterative, and complete. I am no longer limited to showing how a product might work. I can prototype its behavior, examine its architecture, document its decisions, and prepare it for implementation.”
